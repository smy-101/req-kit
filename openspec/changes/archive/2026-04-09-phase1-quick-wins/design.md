## Context

req-kit 当前已有基础的键盘快捷键（Ctrl+Enter 发送、Escape 关闭模态框、Ctrl+W 关闭标签、Ctrl+T 新建标签），但缺少保存、切换标签等高频操作快捷键。UI 只有深色主题（"Midnight Forge"），所有颜色硬编码在 CSS 变量中（`:root`），没有浅色主题，也没有切换机制。集合运行器遇到请求失败时会标记为 failed 并继续执行后续请求，但没有重试机制，偶发网络问题会导致整个运行结果不可信。

技术约束：
- 前端零框架，基于 `store` 事件驱动 + DOM 操作
- CSS 已使用 CSS 变量体系（`--bg-0` 到 `--bg-5`、`--text-0` 到 `--text-4`、`--accent` 等），但只有 `:root` 一套值
- 集合运行器通过 SSE 流推送事件，后端 `RunnerService.run()` 顺序执行请求
- 无构建步骤，无 Tailwind，纯手写 CSS

## Goals / Non-Goals

**Goals:**
- 补全高频键盘快捷键，覆盖保存、标签切换、新建请求等操作
- 基于现有 CSS 变量体系增加浅色主题，支持一键切换并持久化
- 集合运行器支持可配置的请求失败重试

**Non-Goals:**
- 不做快捷键自定义绑定 UI
- 不做主题编辑器或更多主题预设
- 不做单次请求的重试（仅集合运行器场景）
- 不做指数退避或复杂重试策略

## Decisions

### 1. 键盘快捷键 — 扩展现有 keydown 监听器

**选择**：将快捷键匹配逻辑抽取为 `js/utils/shortcuts.js` 中的纯函数 `matchShortcut()`，`app.js` 中的 keydown 监听器调用该函数并分发动作。不引入快捷键管理库。

**理由**：快捷键数量有限（约 10 个），引入库（如 mousetrap）增加依赖但收益不大。抽取纯函数（而非完整模块）既保持了 `app.js` 作为快捷键注册入口的一致性，又使匹配逻辑可独立单元测试，无需 DOM 环境。

**替代方案**：
- 引入 mousetrap/hotkeys 库 → 增加外部依赖，违反零依赖原则
- 全部内联在 app.js 中 → 匹配逻辑与 DOM 副作用耦合，无法独立测试

### 2. 主题切换 — CSS 变量 + `[data-theme]` 属性选择器

**选择**：在 `<html>` 元素上设置 `data-theme="dark"` 或 `data-theme="light"` 属性，用 `[data-theme="light"]` 选择器覆盖 `:root` 的 CSS 变量值。偏好存储在 `localStorage`。

**理由**：现有 CSS 已经全部基于 CSS 变量，只需要提供另一套变量值即可。`data-theme` 属性是 CSS 主题切换的标准模式，无 JS 运行时开销。不需要为每个组件修改样式。

**替代方案**：
- 两套完整 CSS 文件切换 → 维护成本高，样式易不一致
- CSS `color-scheme` + `prefers-color-scheme` → 只能跟随系统，无法手动切换
- JS 动态修改变量值 → 运行时开销，不如纯 CSS 优雅

### 3. 请求重试 — RunnerService 内部循环重试

**选择**：在 `RunnerService.run()` 的请求执行循环中，当请求失败时根据配置进行重试。重试配置通过 API 请求参数传入（`retry_count`、`retry_delay_ms`）。

**理由**：重试逻辑与运行器执行流程紧密耦合，放在 RunnerService 内部最自然。通过请求参数传入而非持久化到数据库，因为重试配置是运行时决策而非请求属性。

**替代方案**：
- 独立 RetryService → 过度抽象，重试逻辑简单
- 数据库存储重试配置 → 增加复杂度，配置是临时的运行参数
- 前端重试（重新发送单个请求）→ 无法利用运行时的变量传递上下文

## Risks / Trade-offs

**[快捷键冲突]** → Ctrl+S、Ctrl+W 等可能与浏览器默认行为冲突。使用 `e.preventDefault()` 拦截，但 Ctrl+S 在某些浏览器中无法完全拦截。如果拦截失败，快捷键不生效但不会破坏功能。

**[浅色主题视觉一致性]** → 需要为所有 CSS 变量定义浅色值，可能需要微调个别组件的样式。通过 CSS 变量覆盖的方式，大部分组件应该自动适配，但可能发现少量硬编码颜色需要修正。

**[重试掩盖真实错误]** → 自动重试可能掩盖间歇性 bug。通过 SSE 事件推送每次重试的状态（`request:retry` 事件），UI 上明确显示重试次数和原因，让用户知晓发生了重试。默认重试次数设为 0（不重试），由用户主动开启。
